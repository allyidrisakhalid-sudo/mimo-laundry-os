# Final Polish Runbook  Mimo Phase 2

## 1) Purpose

This runbook defines the final operator-facing polish pass required before launch-quality claims are made for Mimo Phase 2. The purpose of this pass is to review the product as a connected user experience across public entry, auth, support, and all role portals so that speed, states, copy, spacing, icons, alignment, and overall visual discipline meet the world-class minimal standard.

## 2) Audit Order

Locked audit order:
1) speed and responsiveness
2) empty/error/loading states
3) EN/SW copy consistency
4) spacing/icon/alignment cleanup
5) final cross-role spot check

## 3) Required Screens and Flows to Review

Required screens and flows:
- public home
- public track
- login
- signup
- customer home / wizard / order detail / profile
- driver today / stop detail
- hub intake / processing / dispatch
- affiliate order creation / list / detail / finance
- admin dashboard / orders / operations / finance
- DevAdmin home / diagnostics / tools / activity
- support issue flow
- refund / credit visibility states

Review rules:
- review each surface on its intended device class first
- review mobile-critical surfaces on everyday phone widths first
- review dense operational surfaces with realistic data states where possible
- review route transitions between related screens, not only isolated static views
- include shared support and issue patterns where they affect trust across multiple roles

## 4) Evidence Capture Rules

Locked evidence capture rules:
- capture before/after screenshots for meaningful polish fixes
- capture notes for friction, inconsistency, or visual drift
- capture device context where relevant
- do not mark polish complete without visible evidence

Evidence guidance:
- screenshots should clearly show the problem and the corrected state
- device context should note phone, tablet, or desktop intent where relevant
- notes should describe what was visible, why it mattered, and what changed
- evidence should be retained for any issue that affected speed perception, responsiveness, state quality, copy quality, or visual consistency

## 5) Issue Logging Rules

Locked issue logging rules:
- every polish issue records:
  - screen/flow
  - type:
    - speed
    - responsiveness
    - empty/error/loading
    - copy
    - spacing/icon/alignment
  - severity
  - visible symptom
  - expected polished behavior
- no vague UI needs cleanup logging

Issue logging guidance:
- screen or flow names must be specific enough to locate the issue quickly
- type must reflect the dominant problem category
- severity must follow the locked friction model
- visible symptom must describe what the operator actually saw
- expected polished behavior must describe the launch-ready outcome
- issue logs must remain concrete, reproducible, and honest

## 6) Fix Priority Rules

Locked fix priority rules:
- Critical and Major issues must be fixed before PASS
- Moderate issues should be fixed in this pass where feasible
- Minor issues may remain only if they do not break the world-class minimal gate and are documented honestly

Priority handling rules:
- if a moderate issue appears repeatedly across shared components or critical flows, treat it as a broader risk and reassess severity
- no fix priority decision may hide visible trust problems in core journeys
- no issue may be downgraded just to reach PASS
- launch readiness depends on honest severity treatment, not optimistic labeling

## 7) Re-test Rules

Locked re-test rules:
- after a polish fix, re-check the full affected screen or flow
- if spacing or copy changed in a shared component, spot-check all screens using it
- if a responsive issue changed layout behavior, re-check at all relevant target widths

Re-test guidance:
- do not test only the specific fragment that was edited if the issue affects a larger journey
- re-check supporting states such as empty, loading, and error modes when a shared component changed
- repeat route transition checks where layout or structure changed
- confirm no new visual drift or copy contradiction was introduced while fixing the issue

## 8) Final Pass Summary Rules

Locked final pass summary rules:
- summarize remaining issues by severity
- summarize cross-role polish status
- state clearly whether the product now feels launch-ready and world-class minimal
- no false PASS if visible roughness remains in core flows

Summary guidance:
- the summary must reflect what remains open, not only what was fixed
- cross-role polish status must include public, customer, driver, hub, affiliate, admin, DevAdmin, and support
- launch-ready language may only be used when the final pass shows no critical issues, no unresolved major issues, and no meaningful roughness undermining the standard
- if visible roughness remains in core flows, the result must stay honest and not claim PASS

## 9) Runbook Lock Statement

The final polish runbook is locked as the required operator method for judging whether Mimo Phase 2 is truly launch-ready. The product must be reviewed as one coherent experience across roles, devices, and state conditions, with evidence captured, issues logged precisely, severity judged honestly, and re-tests completed after every meaningful repair. No launch-quality claim is valid until this runbook is executed and the world-class minimal standard is visibly met.
