# SEO and Content Intent  Mimo Phase 2 Public Experience

## 1) SEO Scope for P2.3

P2.3 SEO scope is limited to the essential public routes that explain the service, support discovery, and match page intent cleanly on mobile search. This chapter does not introduce a blog, content hub, city landing page program, or SEO-heavy expansion system.

Locked SEO scope:
- route-level metadata clarity
- page intent clarity
- mobile search snippet readability
- duplicate content prevention
- direct match between search intent and first visible content

## 2) Route Intent Map

### Route: /
- search intent
  - visitors looking for Mimo, laundry service, pickup laundry, or an overview of what the service does
- page purpose
  - explain the service clearly and move the visitor to the correct next step
- indexing intent
  - index
- duplication risk to avoid
  - do not turn home into a tracking utility page, partner recruitment page, or FAQ library

### Route: /track
- search intent
  - returning users looking to track an order quickly
- page purpose
  - provide direct order tracking utility
- indexing intent
  - index
- duplication risk to avoid
  - do not turn tracking into a dashboard, login landing page, or general help center

### Route: /partners
- search intent
  - shop owners or operators looking for partnership information
- page purpose
  - convert qualified affiliate partnership interest
- indexing intent
  - index
- duplication risk to avoid
  - do not reuse the full home narrative or generic customer-ordering content

### Route: /help
- search intent
  - users looking for support, common answers, or issue escalation
- page purpose
  - centralize support guidance and policy summaries
- indexing intent
  - index
- duplication risk to avoid
  - do not duplicate the full legal pages or the full tracking workflow

### Route: /login
- search intent
  - users trying to access their account or portal
- page purpose
  - authenticate existing users and route them correctly
- indexing intent
  - low-value for broad discovery; allow indexing only if it remains clean and useful, but do not optimize it as a marketing page
- duplication risk to avoid
  - do not duplicate signup value propositions or home page service story

### Route: /signup
- search intent
  - new users trying to create an account or start with Mimo
- page purpose
  - create account and begin the approved start flow
- indexing intent
  - index if implemented as a useful clean entry page
- duplication risk to avoid
  - do not duplicate the full landing page persuasion stack

### Route: /terms
- search intent
  - users looking for service terms or agreement details
- page purpose
  - present basic service terms in readable form
- indexing intent
  - index
- duplication risk to avoid
  - do not duplicate help FAQs or marketing content

### Route: /privacy
- search intent
  - users looking for privacy information and data handling details
- page purpose
  - explain how user and customer data is handled
- indexing intent
  - index
- duplication risk to avoid
  - do not duplicate terms language unnecessarily or add promotional content

### Route: /refund-policy
- search intent
  - users looking for refund handling and issue resolution expectations
- page purpose
  - explain refund and issue handling clearly
- indexing intent
  - index
- duplication risk to avoid
  - do not bury refund guidance inside help only or duplicate long legal text elsewhere

## 3) Metadata Principles

Locked metadata rules:
- one clear title intent per route
- one clear description intent per route
- no duplicated metadata patterns across multiple pages
- public pages must say what the page is, not generic slogans

Additional metadata rules:
- titles should lead with the route purpose before brand decoration
- descriptions should describe the exact page task or value
- avoid empty premium language that hides page meaning
- home metadata should explain the service plainly
- track metadata should describe order tracking directly
- partner metadata should describe partnership opportunity directly
- help metadata should describe support and answers directly
- legal metadata should identify the exact policy page clearly

## 4) Content Duplication Guard

Locked ownership model:
- home owns the service story
- track owns tracking utility
- partners owns partner conversion
- help owns support guidance
- login/signup own auth entry
- legal pages own policy detail

Additional duplication controls:
- no route should repeat another routes full headline stack and body structure
- support snippets on home and track must stay short and route to /help when needed
- policy summaries on /help must stay summary-only and link out
- login and signup must remain utility pages, not mini-marketing sites
- footer repetition must stay navigational, not explanatory

## 5) Mobile Search Experience Rules

Locked mobile search rules:
- titles and descriptions must stay understandable on mobile search snippets
- first visible content must confirm the search intent quickly
- no slow hero clutter before page meaning is visible

Additional mobile rules:
- page header must state the page function immediately
- the main CTA or main task should appear near the top on mobile
- decorative imagery must not push the real page purpose below the fold
- pages should load as concise, scan-friendly destinations, not long theatrical intros

## 6) SEO Lock Statement

The P2.3 public SEO model is locked to intent clarity, route ownership, duplication control, and mobile-first comprehension. Every public route must match a distinct user/search intent, confirm that intent immediately in visible content, and avoid borrowing the full job of another route.
