# Driver Mobile Behavior Spec  Mimo Phase 2

## 1) Mobile Principles

Locked mobile principles:
- one-hand usable
- task-first
- minimal taps
- big proof actions
- route clarity before detail density
- poor-network-safe messaging
- no desktop carryover clutter
- fast daily repetition

## 2) Bottom Tab Behavior

Locked bottom tabs:
- Today
- Tasks
- Profile

### Bottom Tab Rules
- Today is the default execution home
- Tasks provides the wider list of assigned work
- Profile contains availability, help, and logout
- tabs must remain simple, stable, and easy to hit with one hand
- the bottom tab bar must not add extra destinations beyond the approved navigation
- stop detail lives above the tab level and is opened from Today or Tasks

## 3) Today Screen Mobile Behavior

- next task visible immediately
- grouped tasks stack in execution order
- cash and alert blocks remain secondary but visible
- no horizontal scrolling
- no dense admin-like data pattern
- today summary remains compact
- the next task hero card must sit near the top and remain visually dominant
- grouped lists must stay scan-friendly with generous tap areas
- primary CTA must remain easy to reach with one thumb

## 4) Tasks Screen Mobile Behavior

- list remains simple
- search/filter only if truly needed and light
- task rows remain tap-friendly
- completed vs pending states remain visually clear
- task rows must prioritize action clues over metadata density
- status, proof, and cash indicators must stay compact and readable
- no table layout or desktop carryover list density

## 5) Stop Detail Mobile Behavior

- proof action sits above the fold
- buttons large and easy to hit
- issue/help path visible but secondary
- cash prompt visible when needed
- no long metadata wall before the action
- stop summary must stay concise
- proof flow must sequence clearly if more than one input is required
- retry, pending, or synced states must appear close to the action area

## 6) Profile Mobile Behavior

- availability and account controls clear
- help/logout easy to find
- no bloated settings screen
- profile must stay operational, not preference-heavy
- support entry points must remain easy to access without distracting from daily task flow

## 7) Network and Performance Rules

- mobile screens must remain readable under poor connection
- layout stability must hold during retries/sync states
- no heavy media or decorative load in the driver portal
- task actions must feel fast even when server response is delayed slightly
- signals for pending/synced/failed must remain compact and consistent
- loading behavior must not jump key controls around the screen
- the portal must prioritize action reliability over visual complexity
- all core task states must remain understandable on smaller devices without zooming or horizontal scroll

## 8) Mobile Lock Statement

The driver portal mobile behavior is locked as a one-hand, task-first system optimized for fast daily repetition, large proof actions, calm network-state messaging, and minimal execution friction. It must not inherit desktop clutter or compromise the clarity of the drivers next action.
